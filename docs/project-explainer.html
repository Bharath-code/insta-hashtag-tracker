Instagram Hashtag Tracker

An ingestion pipeline that pulls Instagram media for the matcha hashtag from the Meta Graph API, stores metadata in Postgres, copies assets to durable storage, and serves one cursor-paginated read API.

What this product does

Keep a durable, queryable feed of Instagram posts tagged #matcha, refreshed automatically, without depending on Meta's short-lived CDN URLs.

Ingest

Fetch from Meta

Worker pulls top_media once and recent_media every 3 hours via Graph API, paginated 50/page up to 500 items.

Persist

Upsert + copy assets

Metadata lands in Postgres with Meta media ID as PK. Binary assets are copied to S3 or ./storage/ because CDN URLs expire.

Serve

Read API only

Stateless Express exposes GET /hashtags?limit&cursor — newest posted_at first, keyset pagination, no writes.

Why two processes? Ingestion is bursty (hundreds of downloads per sync). The API stays read-only and scalable on its own so a heavy sync never slows feed responses.

Design rule

Only two interfaces

Queue and Storage each have AWS + local drivers selected by QUEUE_DRIVER / STORAGE_DRIVER.

No repository-over-ORM, no DI container, no speculative interfaces. Docs explicitly reject premature abstraction common in take-homes.

Step 1 of 8
Idempotency is the product guarantee. SQS is at-least-once. Replaying a job re-upserts the same media IDs and only uploads rows still missing storage_key. Retries finish partial work; they don't create duplicates.

Index

(posted_at DESC, id DESC)

Matches the keyset query exactly: WHERE (posted_at, id) < (cursor). Feed sort is Instagram post time, not ingestion time.

Upsert

ON CONFLICT (id) DO UPDATE

Refreshes like/comment counts. Same post in top + recent → one row. Concurrent workers safe without read-before-write.

Left out on purpose: owner/username (hashtag API doesn't return it), carousel children as separate rows, raw JSONB payload. Scope stays tight for the assignment.

Scale path (already accommodated)

  • More hashtags — schema ready (FK on media)
  • Worker → Lambda — handler is already (job) ⇒ Promise
  • API horizontal scale — stateless reads only
  • Assets via CloudFront in front of S3
  • Hashtag ID cached — Meta 30 searches / 7 days limit
  • Meta client retries 429/5xx (3× backoff); 4xx fail-fast
  • Asset failures log & skip — never kill the batch
  • Env validated with zod at boot — fail fast
